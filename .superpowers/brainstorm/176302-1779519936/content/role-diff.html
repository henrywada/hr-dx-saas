育成ジャーニーボード — 役割別の表示・操作の違い

「共有」の意味は「同一URLを共有する」ではなく「同じデータ・同じレイアウトを見ながら対話できる」です。
URLは上司と従業員で別々ですが、表示する情報・デザインは共通。ボタン類だけ役割で出し分けします。
アクセス権は skill_approvers テーブルを使ってサーバー側で厳密にチェックします。

🧭 上司ログイン時（approver_id = 自分）
/skill-approvals/journey/[employeeId]
表示される情報
担当メンバーの氏名・目標・進捗表示
メンバーの自己評価・メモ表示
メンバーからの相談（SOS）表示
他の従業員のデータ完全非表示・403
操作できる機能
目標職種・期限を提案する可
マイルストーンを提案・変更する可
アドバイスコメントを送る可
相談（SOS）に返答する可
要件達成を承認する可
自己評価を記入する不可
SOSボタン（相談を送る）非表示
ナビゲーション
← チーム育成一覧 に戻る表示
← マイスキルに戻る非表示
👤 従業員ログイン時（employee_id = 自分）
/my-skills/journey （employeeId は自動的に自分）
表示される情報
自分の目標・進捗表示
自分の自己評価・メモ表示
上司からのアドバイス表示
他の従業員のデータ完全非表示・URLで指定不可
操作できる機能
上司からの目標提案を承認する可
マイルストーン変更提案に同意する可
自己評価・メモを記入・更新する可
上司に相談する（SOSボタン）可
目標・マイルストーンを自分で提案不可（上司のみ）
アドバイスを送る不可
ナビゲーション
← チーム育成一覧 に戻る非表示
← マイスキルに戻る表示
skill_approvers によるアクセス制御フロー（サーバー側）
上司がアクセス → skill_approvers
WHERE approver_id = 自分 → 担当employee_id一覧を取得
URLの:employeeId が一覧に含まれる?
✅ YES → 上司ビューで表示 | ❌ NO → 403 アクセス拒否
従業員がアクセス → /my-skills/journey のemployeeIdは session の employee_id に固定 → 常に自分のデータのみ
「目標変更」の提案フロー（臨機応変に変更できる仕組み）
上司がマイルストーン変更を提案 → ステータス: 提案中 として保存 → 従業員が「同意する」ボタンで確定。
双方が合意するまで変更は pending 状態のままで、現在の計画は継続されます。
これにより 一方的な変更を防ぎ、常に両者の合意で進む設計になります。
