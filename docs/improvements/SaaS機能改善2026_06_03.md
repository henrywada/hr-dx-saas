# HR-DX SaaS 機能改善計画 2026-06-03

作成日：2026-06-03
参照：docs/improvements/課題TOP10.md、src/features/（開発済み機能）

---

## 1. ターゲットペルソナと基本方針

### ペルソナ

従業員100〜1000名規模の中小企業における経営者・人事責任者。

| ペルソナ | 最大の関心事 |
|---|---|
| 経営者 | 離職・採用コスト削減、組織の健全性を数字で掴む、後継者・幹部候補の見える化 |
| 人事責任者 | 労務コンプライアンス遵守、評価・採用ワークフローの効率化、現場の本音把握 |

### 改善の基本方針

- **データを繋げる** — 個別機能は揃っている。分断データを横断可視化することが最大の価値
- **予兆を掴む** — 離職・法令違反・評価遅延をリアルタイム検知しアラート。後手から先手へ
- **シンプル&すぐ使える** — 1画面で状況が分かる設計。操作は「見る・送る・承認する」3動作以内
- **工数対効果優先** — 既存データを活用するUIから着手。新テーブルが必要な機能は後回し

## 2. 現状機能の棚卸し

| 機能区分 | 開発済み features | 備考 |
|---|---|---|
| 採用管理 | job-postings, candidate-pulse, offer-validation, recruitment-ai, market-analysis | 個別機能は揃う |
| 勤怠・労務 | attendance, overtime, qr-punch, telework | データあり |
| ストレスチェック | stress-check, high-stress, high-stress-followup, pulse-stress | 法令対応済 |
| パルスサーベイ | survey, questionnaire, echo-template | 複数ソース分散 |
| 人事評価 | evaluation, global-evaluation-templates | ワークフロー未自動化 |
| スキル育成 | skill-map, skill-portal, global-skill-templates, e-learning | 連携不足 |
| AI活用 | ai-workplace-improvement, inquiry-chat | 個別機能のみ |
| オンボーディング | onboarding | 退社フローなし |
| 組織管理 | organization, division-establishments | — |
| ダッシュボード | dashboard, saas-dashboard | 横断ビューなし |

### 最大のギャップ（3点）

1. **横断ダッシュボードがない** — 各機能のデータが別画面に分散。経営者が「今の組織状態」を一目で見られない
2. **予兆アラートがない** — 離職リスク・残業超過・評価遅延が起きてから気づく
3. **ワークフロー自動化不足** — 評価リマインダー・入退社チェックリストが手作業

## 4. フェーズ別 機能改善計画

---

### Phase 1 — クイックウィン（目安 1〜2ヶ月）

既存テーブルのデータを活用するUI中心。新テーブル追加は最小限。工数小・効果大。

---

#### P1-A　採用ファネルダッシュボード

対応課題：#1 採用難

**ペルソナが得るもの**

- 経営者：今週何人面接して内定を出しているかが1分で分かる
- 人事：放置候補者をゼロにできる。担当者への催促が自動化される

**機能**

- 応募→書類選考→面接→内定→入社 のファネル数値表示
- N日以上アクションなし候補者のハイライト（N日は設定可能）
- 担当者別タスク残件数バッジ
- 内定辞退率の推移グラフ

実装先：`src/features/job-postings/` 拡張
ルート：`/adm/recruit/funnel`
工数目安：3〜5日

---

#### P1-B　労務コンプライアンスダッシュボード

対応課題：#5 労務管理・法令対応

**ペルソナが得るもの**

- 経営者：法令違反リスクがある社員が今月何人いるか一目で分かる安心感
- 人事：月末の集計作業ゼロ。毎日画面を見るだけでアラート確認できる

**機能**

- 残業アラート一覧（月45h・年360hライン）
- 有休取得義務進捗（年5日未達リスク者リスト）
- 36協定特別条項対象者カウント
- 部署別ヒートマップ（残業率・有休取得率）

実装先：`src/features/overtime/` + `src/features/attendance/` 結合
ルート：`/adm/labor-compliance`
工数目安：3〜5日

---

#### P1-C　評価ワークフロー自動化

対応課題：#3 人事評価制度の形骸化

**ペルソナが得るもの**

- 経営者：評価サイクルが予定通り終わる。詰まっている箇所が一目で分かる
- 人事：催促メールの手作業がなくなる。未提出者を一括催促できる

**機能**

- 評価期限N日前・当日・超過のリマインダー自動送信
- フェーズ進捗バー（目標設定→中間→最終→承認）
- 未提出者リスト＋一括催促ボタン
- 差し戻し通知（評価者→被評価者）

実装先：`src/features/evaluation/` 拡張（既存タブに追加）
工数目安：4〜6日

---

#### P1-D　横断KPIダッシュボード（経営層向け）

対応課題：#9 データ分断・可視化不足

**ペルソナが得るもの**

- 経営者：月次役員会議資料を自動生成できる。人事の状況を数字で語れる
- 人事：各ツールを横断して集計していた作業がなくなる

**機能**

- 採用KPI：応募数・選考通過率・充足率
- 定着KPI：離職率・平均在籍年数
- 生産性KPI：平均残業時間・有休取得率
- エンゲージメントKPI：パルスサーベイ平均スコア・高ストレス率
- 育成KPI：スキルギャップ率・研修完了率
- PDF／CSVエクスポートボタン

実装先：`src/features/dashboard/` 拡張
ルート：`/adm/hr-kpi`
工数目安：5〜8日

## 5. 新機能の開発（追加提案）

既存機能の改善では対応できない、ペルソナ課題を解決するための新機能。

---

#### NEW-1　AIチャット型人事相談アシスタント

課題背景：人事担当者が「この場合、有休はどう計算するか」「この評価コメントは適切か」を調べるのに毎回時間がかかる。

**ペルソナが得るもの**

- 人事：法令・社内規程・過去事例に基づいた即答が得られる。調査時間を大幅削減
- 経営者：人事知識のない担当者でも正確な対応ができる体制が作れる

**機能**

- 社内規程・就業規則をRAGで参照するチャットUI
- 労務計算（残業代・有休消化日数）の自動回答
- 評価コメントの添削・改善提案（AI）
- 過去の類似ケース検索

実装先：`src/features/hr-assistant/`（新設）、`inquiry-chat/` を基盤として拡張
工数目安：10〜15日

---

#### NEW-2　目標管理（OKR / MBO）機能

課題背景：評価制度はあるが、日常の目標設定と進捗管理がExcelで行われており、評価と連動していない。

**ペルソナが得るもの**

- 経営者：会社目標から部門・個人目標まで一気通貫で見える。「戦略が現場に落ちているか」が分かる
- 人事：評価と目標が連動し、「評価のための評価」から脱却できる

**機能**

- 会社→部門→個人のOKRツリー表示
- 目標進捗の週次チェックイン（5段階スライダー＋コメント）
- 期末評価との自動連動
- 目標達成率のダッシュボード

新テーブル：`objectives`、`key_results`、`checkins`
実装先：`src/features/okr/`（新設）
ルート：`/adm/okr`
工数目安：12〜18日

---

#### NEW-3　採用ブランディング支援（求人票AI生成）

課題背景：知名度で大企業に勝てない中、自社の魅力を言語化した求人票が書けず、応募数が増えない。

**ペルソナが得るもの**

- 経営者：採用広告費をかけなくても、自社の魅力が伝わる求人票が作れる
- 人事：求人票作成にかかる時間を1/3以下に短縮できる

**機能**

- 会社情報・職種・文化を入力するとAIが求人票を自動生成
- 複数バリエーション生成（媒体別：indeed / LinkedIn / ハローワーク等）
- 既存 `job-postings` への直接投稿
- 類似企業との差別化ポイント提案

実装先：`src/features/job-postings/` 拡張（AIライティングモジュール追加）
工数目安：6〜10日

---

#### NEW-4　リファラル採用管理

課題背景：社員紹介採用（リファラル）が最も採用コストが低く質が高いが、仕組みがなく運用されていない。

**ペルソナが得るもの**

- 経営者：採用単価を下げながら、マッチング精度の高い採用ができる
- 人事：リファラルの推薦状況・報奨金支払いを一元管理できる

**機能**

- 社員が求人票を見て知人を推薦するフォーム
- 推薦→選考→採用の進捗追跡
- 報奨金の支払い管理（採用確定時トリガー）
- 推薦件数ランキング（社内のモチベーション向上）

実装先：`src/features/referral/`（新設）
ルート：`/adm/referral`
工数目安：8〜12日

## 7. UI設計の共通原則

- **1画面1メッセージ** — その画面を開いたら「今何が問題か」が3秒で分かる
- **アクションは3つまで** — 「詳細を見る」「催促する」「エクスポート」が上限
- **色で状態を伝える** — アラートは赤、安全は緑、注意は黄。テキストだけに頼らない
- **ドリルダウン設計** — 全体→部署→個人の順番で詳細へ進む
- **既存スタイルに準拠** — `docs/ui/admin-card-and-table.md` のスタイルを維持
- **必須ファイル配置** — 各ルートに `loading.tsx` と `error.tsx` を必ず配置

## 8. 実装時の技術指針

データアクセスパターン（CLAUDE.md準拠）：

```
page.tsx（Server Component）
  → src/features/[domain]/queries.ts   # SELECT のみ
  → Client Component に Props で渡す

Client Component
  → src/features/[domain]/actions.ts  # Server Actions（INSERT / UPDATE / DELETE）
```

Phase 2で追加する主要テーブル（最小構成）：

```sql
-- 離職リスクスコア（P2-A）
CREATE TABLE turnover_risk_scores (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL REFERENCES tenants(id),
  employee_id UUID NOT NULL REFERENCES employees(id),
  risk_score  INTEGER NOT NULL,   -- 0〜100
  risk_level  TEXT NOT NULL,      -- 'high' | 'medium' | 'low'
  score_factors JSONB,            -- スコア計算要因の内訳
  calculated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
ALTER TABLE turnover_risk_scores ENABLE ROW LEVEL SECURITY;

-- 1on1記録（P2-C）
CREATE TABLE one_on_one_records (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL REFERENCES tenants(id),
  manager_id  UUID NOT NULL REFERENCES employees(id),
  member_id   UUID NOT NULL REFERENCES employees(id),
  scheduled_at TIMESTAMPTZ NOT NULL,
  completed_at TIMESTAMPTZ,
  theme       TEXT,
  notes       TEXT,
  next_action TEXT
);
ALTER TABLE one_on_one_records ENABLE ROW LEVEL SECURITY;
```
